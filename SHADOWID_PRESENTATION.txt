                 PAGE 3: CREATE SHADOWID (Identity Creation)

[User clicks "Create ShadowID"]

"Creating a ShadowID is the most critical step. Here's how it works:

STEP 1: ATTRIBUTE SELECTION

Users see all available attributes from our standard schema. Examples include:
- Age Range (e.g., '18-25', '26-35', '36-45', '46-55', '56-65', '66+')
- Jurisdiction (e.g., 'United States', 'European Union', 'Asia-Pacific')
- Professional Title (e.g., 'Software Engineer', 'Doctor', 'Lawyer')
- Educational Background (e.g., 'Bachelor''s Degree', 'Master''s Degree')
- DAO Member Status
- Organizational Affiliation

For EACH attribute they claim, they:
1. Enable it (checkbox)
2. Enter a value from the predefined options
3. This becomes part of their identity commitment

Why this matters: These are the facts about themselves they want to be verified. 
They're not yet proven—they're claims. The peer endorsement system verifies them.

STEP 2: COMMITMENT GENERATION

Behind the scenes, a sophisticated process happens:

1. The system takes all their selected attributes and values
2. Creates a cryptographic hash combining all this data
3. Generates a mathematical 'commitment'—a unique fingerprint of their identity
4. This commitment is registered on the Aleo blockchain

The blockchain now knows: 'This wallet address is associated with this commitment.'

But the blockchain knows NOTHING about the actual attributes or their values. 
Only the commitment hash exists on-chain.

STEP 3: LOCAL ENCRYPTION & STORAGE

The system generates:
- Commitment Hash: A checksum-verified version with format: 'XXXX-YYYYYYYYYY...'
  This is their public identity identifier they can share
- Encryption Key: Derives from their wallet
- All attribute data is encrypted locally and stored in localStorage

The commitment hash is displayed:
'Your ShadowID: 927B-0714FE5AEB7CE97358F300578FB38EI'

Copy button lets them save it.

STEP 4: CONFIRMATION SCREEN

Success state shows:
- ✓ ShadowID Created Successfully! 🎉
- 'Your zero-knowledge identity is now active on the Aleo testnet.'
- Their commitment hash clearly displayed
- QR code of their commitment for sharing
- Options to proceed to their identity page

BLOCKCHAIN VERIFICATION:
The system checks if this wallet already has an account on the blockchain. 
If yes, it recovers the previous commitment. If no, it registers the new one.

This enables account recovery—if they connect from a different device with the 
same wallet, they automatically recover their ShadowID.

SUBSCRIPTION LIMITS:
Users have limits on how many attributes they can activate based on a 
subscription tier:
- Free: 3 attributes
- Premium: 10 attributes
- Pro: Unlimited

If they exceed their limit, a subscription modal appears offering upgrades.

ANTI-SYBIL PROTECTION:
The system tracks account creation attempts and rate-limits users to prevent 
creating multiple identities. Each wallet address can only create one identity 
per time period."

"This is the heart of the decentralized reputation system.

PROCESS:

Step 1: INPUT PHASE

2. Select an Attribute
   A dropdown showing all standard attributes:
   - Age Range

   - Etc.

VALIDATION:
The system checks several conditions BEFORE allowing the endorsement:

1. Self-Endorsement Check
   'You cannot endorse your own attributes'
   Their own commitment hash is detected and rejected.

2. Account Status Check
   'You must create a ShadowID before endorsing peers'
   Users need their own identity to endorse others.

3. Format Validation
   Ensures the commitment hash is valid format:
   - New format: 'XXXX-YYYYYYYYYY...' (checksum-hash)
   - Legacy hex format: '0x...' or pure hex

4. Flagging Check (Anti-Sybil)
   'Cannot endorse: this account has been flagged by administrators'
   If the target account is flagged as a Sybil or fraudulent, 
   the endorsement is blocked entirely.

5. Rate Limiting
   Prevents spam endorsements. Each user has a rate limit on how many 
   endorsements they can give per day.

Step 2: CONFIRMATION
Shows:
- Their commitment hash
- Target commitment hash
- Attribute being endorsed
- Summary: 'You are confirming that [Target Hash] genuinely has [Attribute]'

Step 3: BLOCKCHAIN TRANSACTION
The system calls the Aleo smart contract:
execute_transaction('endorse_attribute', {
  target_commitment: [hash],
  attribute_id: 'jurisdiction',
  endorser_address: [wallet]
})

The blockchain records this permanently. It can never be deleted or falsified.

Step 4: CONFIRMATION SCREEN
✓ Endorsement Successful

'You endorsed the Age Range attribute for commitment 927B-0714FE...'

Shows:
- Transaction hash for immutability proof
- The endorsement is now live on the blockchain
- Option to endorse another attribute or go back

IMPORTANT FEATURES:

Anonymous But Traceable:
- The endorsement is recorded on blockchain
- It shows your wallet address endorsed someone
- But endorsements aren't tied to your real identity—just your wallet
- Creates pseudonymity with transparency

Trust Network Formation:
As users endorse each other, a network forms:
- Users with many diverse endorsers gain high Shadow Scores
- Users who endorse high-credibility people build trust themselves
- Sybils (fake accounts) are isolated because they can't get diverse endorsements

Spam Prevention:
- Rate limits prevent one account endorsing many others rapidly
- Flagged accounts can't be endorsed
- Self-endorsement is impossible"

         PAGE 6: SELECTIVE DISCLOSURE (Generate Proof QR Codes)
=================================================================================

[User goes to create a selective disclosure proof]

"This is where zero-knowledge proofs become practical.

SCENARIO:
'An online community wants to verify you're over 18 before letting you into 
an adult section. You need to prove your age without revealing your exact 
age or any other information.'

THE SOLUTION: Selective Disclosure Proofs

INTERFACE:

Left Side: Available Attributes
Shows all attributes the user activated when creating their ShadowID.
They're displayed as toggleable checkboxes:

☐ Age Range
☐ Jurisdiction  
☐ Professional Title
☐ Educational Background
☐ DAO Member Status

Right Side: Attribute Values
For each selected attribute, they must enter the specific value they're 
proving. For example:

Age Range: [26-35] (dropdown select)
Jurisdiction: [United States] (dropdown select)

This is CRITICAL: They're saying 'I'm claiming this specific value for this 
attribute, and I'm about to prove it.'

THE PROOF GENERATION PROCESS:

1. User selects which attributes to disclose
2. Enters the specific values they claim
3. System creates a zero-knowledge proof showing:
   'This user (identified by their commitment hash) genuinely has these 
    attribute values'
   
   WITHOUT showing:
   - Any other attributes
   - Their real name
   - Their wallet address
   - Any identifying information

4. System generates a QR code encoding the proof
5. The QR includes:
   - The proof itself (cryptographic mathematical verification)
   - Their commitment hash
   - The attributes they're proving
   - Timestamp and expiration (default 7 days)
   - A nullifier (one-time-use code to prevent replay attacks)

EXPIRATION & SECURITY:

Each QR has an expiration time. After expiration:
- The QR code becomes invalid
- The verifier won't accept it
- User must generate a new QR

This prevents:
- Stolen QR codes being used indefinitely
- Proof leakage weeks later
- Unauthorized access

NULLIFIER (One-Time Use):
Each QR has a unique nullifier—a session identifier. The system tracks which 
nullifiers have been used. If someone tries to use the same QR twice:
- First use: ✓ Accepted
- Second use: ✗ Rejected 'This proof has already been verified'

DOWNLOAD & SHARING:
Users can:
- Download QR as an image
- Display QR on screen
- Email QR code
- Share QR link

The other party simply scans the QR to verify the proof.

DEMONSTRATION:
'Age Range selective disclosure:
- System proves: User has Age Range = 26-35
- It does NOT prove or reveal: Jurisdiction, Title, Name, Address
- Verifier can be confident: User is indeed 26-35
- Privacy maintained: Everything else remains hidden

This is zero-knowledge cryptography in practice.'

COMMON USE CASES:
1. Age verification (gaming, adult sites)
2. Professional credential verification (job applications)
3. Jurisdictional restrictions (financial products)
4. Educational background proof (university admissions)
5. DAO membership verification (exclusive communities)"
